# Internationalization (i18n)

`price-tracker-bot` ships English (`en`, source language) and Italian (`it`) message catalogs. Locale resolution auto-detects the user's preferred language from Telegram and falls back to a server default.

> **Status**: i18n infrastructure (Babel + gettext) lands in Plan 3 F5 (T14-T23). The mechanism described below is what authors and translators will use once F5 ships. Until then, the bot uses inline strings.

## Supported locales

| Code | Language | Status                  |
| ---- | -------- | ----------------------- |
| `en` | English  | Source language         |
| `it` | Italian  | Full translation        |

Adding a new locale follows the workflow in [Adding a new translation](#adding-a-new-translation) below.

## Runtime resolution chain

When a user sends a command, the bot picks a locale by walking this chain:

1. **Telegram per-user** — `update.effective_user.language_code` (the language the user has set in their Telegram client). Mapped to one of the supported codes when there's a match (e.g. `it_IT` → `it`).
2. **Server default** — the `LANG` environment variable (default `en`). Useful for self-hosted instances where users haven't set a Telegram language or sent one that isn't supported.
3. **Hard fallback** — `en`. Always available; the source language can never be unmapped.

If a string is missing from a locale's `.po` file, the runtime falls through to the source string (English text passes verbatim).

## File layout

```
src/price_tracker/locale/
├── en/LC_MESSAGES/
│   ├── messages.po   # source strings + EN identity translation
│   └── messages.mo   # compiled binary (committed)
└── it/LC_MESSAGES/
    ├── messages.po   # IT translation
    └── messages.mo   # compiled binary (committed)
```

Both `.po` (human-editable) and `.mo` (compiled, machine-readable) files are committed to the repository. The compiled `.mo` is what gettext loads at runtime; the `.po` is what translators edit.

A `babel.cfg` at the repo root tells `pybabel extract` which files to scan for translatable strings (Python source files and any Jinja templates if added later).

## Marker functions

In application code, wrap user-facing strings with one of two markers:

```python
from price_tracker.bot.messages import _, ngettext

# Singular
await update.message.reply_text(_("Welcome to price-tracker-bot!"))

# Plural-aware
n = len(products)
text = ngettext(
    "Tracking %(count)d product.",
    "Tracking %(count)d products.",
    n,
) % {"count": n}
```

`_()` is the gettext convention for singular lookup. `ngettext()` returns the right plural form based on the locale's plural rule. Both fall through to the source string if the catalog has no translation.

## Adding a new translation

To add (e.g.) a German translation:

```bash
# 1. Extract source strings into the template (.pot)
pybabel extract -F babel.cfg -k _ -k ngettext -o messages.pot src/

# 2. Initialize a new locale catalog
pybabel init -i messages.pot -d src/price_tracker/locale -l de_DE

# 3. Translate the strings in:
#    src/price_tracker/locale/de_DE/LC_MESSAGES/messages.po
#    Replace each empty msgstr "" with the German translation.

# 4. Compile to binary
pybabel compile -d src/price_tracker/locale

# 5. Register the locale code in src/price_tracker/bot/messages.py
#    by adding "de" to the _AVAILABLE constant.

# 6. Commit both .po and .mo
git add src/price_tracker/locale/de_DE/LC_MESSAGES/messages.{po,mo} src/price_tracker/bot/messages.py
git commit -m "i18n(de): add German translation"
```

The pre-commit `pybabel-compile-check` hook (F5) verifies that every `.po` has a matching, up-to-date `.mo` before letting the commit through.

## Updating translations after string changes

When you add or modify a `_("...")` or `ngettext(...)` call in source code, the `.po` files need to be re-synchronized:

```bash
# 1. Re-extract template
pybabel extract -F babel.cfg -k _ -k ngettext -o messages.pot src/

# 2. Merge new/changed strings into existing locale catalogs
pybabel update -i messages.pot -d src/price_tracker/locale

# 3. Translate any new entries (msgstr "") in each locale's messages.po

# 4. Compile
pybabel compile -d src/price_tracker/locale

# 5. Commit
git add src/price_tracker/locale
git commit -m "i18n: update catalogs for source string changes"
```

Stale `.po` files (where the template has changed but the catalog hasn't been updated) will trigger a CI failure in `security.yml` / `ci.yml` (Plan 3 F6).

## Pluralization rules

Babel handles per-language plural counts automatically based on the locale's CLDR plural rule. Translators see one or more `msgstr` entries per `ngettext` call:

| Locale | Plural forms | Example                                |
| ------ | ------------ | -------------------------------------- |
| `en`   | 2            | `msgstr[0]` (n=1), `msgstr[1]` (other) |
| `it`   | 2            | same as English                        |
| `ru`   | 3            | one / few / many                       |
| `ar`   | 6            | zero / one / two / few / many / other  |

`pybabel init` populates the right number of `msgstr[i]` entries for the locale automatically — translators just fill them in.

## Testing strategy

- **Catalog coverage**: every translated string has both an EN source entry and a translation in every supported locale's `.po`. Untranslated strings are left as the empty msgstr and produce the English fallback at runtime.
- **`.po` ↔ `.mo` consistency**: pre-commit `pybabel-compile-check` hook (F5) runs `pybabel compile --statistics` and rejects commits where any `.mo` is stale relative to its `.po`.
- **CI gate**: the `ci.yml` workflow (F6) re-extracts the template and diffs against the committed `.po` template — a non-empty diff means a developer added/changed a string without re-syncing.
- **Runtime smoke**: the test suite includes parametrized fixtures that load each locale and assert representative strings render in the target language.
